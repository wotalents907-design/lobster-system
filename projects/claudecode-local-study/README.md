# Claude Code 本地实现专题

## 目标

研究：我们是否能在本地实现一个可用的“Claude Code 类系统”，并明确：

1. 能做到什么程度
2. 不能直接复刻的部分是什么
3. 最现实的本地落地路径是什么
4. 应该基于现有 OpenClaw 体系扩建，还是单独做一个轻量 Claude Code-like harness

## 当前结论（第一版）

- **完全复刻官方 Claude Code：不现实，也没必要。**
- **实现一个本地可用、Claude Code 风格的 coding-agent/harness：可行。**
- **最佳路径不是“照着 leaked repo 生抄”，而是“借鉴其架构思想 + 结合我们现有 OpenClaw 工作流 + 只做真正需要的功能”。**

## 先研究的四个问题

### Q1. 本地实现的最小闭环是什么？
至少应具备：
- 会话循环（agent loop）
- 工具调用
- 文件读写
- shell 执行
- 任务状态恢复
- 权限/风险分层
- 记忆与恢复入口

### Q2. 哪些 Claude Code 特性值得借鉴？
- CLI 驱动的 agent loop
- 工具清单与权限分层
- 子代理/子任务隔离
- 上下文压缩
- 会话恢复与任务延续
- 技能/提示词/工作流外挂化

### Q3. 哪些部分不值得照搬？
- 逆向恢复仓里的 shim/fallback 垃圾层
- 与私有 Anthropic 基础设施深耦合的部分
- 为了兼容原工程而存在的历史包袱

### Q4. 我们的现实落地方案是什么？
优先考虑两条路：

#### 路线 A：在 OpenClaw 上做 Claude Code-like 模式
优点：
- 已有工具系统
- 已有记忆体系
- 已有会话/子代理框架
- 已有任务恢复底座
- 开发成本低

缺点：
- 会继承 OpenClaw 当前的部分结构限制
- 不一定能获得“原版 Claude Code”的交互手感

#### 路线 B：单独做一个本地 coding harness
优点：
- 架构更干净
- 可以按我们自己的思路设计
- 更容易做“专供本地虾/云端虾”的定制版本

缺点：
- 需要自己补更多基础设施
- 会和现有 OpenClaw 形成双系统维护成本

## 当前倾向

当前更倾向：

**先走路线 A：基于 OpenClaw 做一层 Claude Code-like coding mode。**

原因：
- 我们已经有可工作的主系统，不该为了追“原版感”再平地起一套完全新内核。
- 真正有价值的是任务恢复、工具分层、子代理、CLI 工作流，而不是执着于一模一样的源码实现。
- 等路线 A 跑顺，再判断是否值得抽离为单独 harness。
