# HANDOFF.md

## 双虾协作规则

### 一、总原则
- 云端虾不是唯一主脑；它是**常驻协调主脑**。
- 本地虾不是纯执行节点；它是**执行主脑 / 收口主脑**。
- 任务分发必须留痕。
- 结果回传必须结构化。
- 谁更适合把事真正做完，谁主导该任务；不是谁名义更高谁主导。

### 二、权责分工
#### 云端虾优先负责
- 长时在线接球
- 跨场景承接
- 任务分流
- 进度跟踪
- 提醒催办
- 面向用户做阶段性汇总

#### 本地虾优先负责
- 本地浏览器登录态相关操作
- 本机文件、本机软件、本机环境
- 工作区文件、规则、任务系统、自动化落地
- 需要强执行、强收口、强交付的任务
- 依赖本地权限、本地状态的执行任务

### 三、谁有主导权
- 需要长期在线盯住的任务：云端主导
- 需要实际落地推进完成的任务：本地主导
- 需要本地登录态/浏览器/文件权限：本地主导
- 需要跨入口接球、先稳住用户体验：云端主导
- 无明确分工前，先判断“谁完成成本更低、完成质量更高”，再定主导

### 四、每次交接必须包含
1. task_id
2. 目标
3. 为什么交接给对方
4. 步骤
5. 输出物
6. 当前状态
7. 是否需要人工确认
8. 卡点/风险
9. 下一步建议

### 五、状态定义
- todo：待执行
- doing：执行中
- blocked：卡住
- done：已完成
- failed：失败

### 六、交接最小模板
```md
- task_id:
- 目标:
- 为什么交接给对方:
- 步骤:
- 输出物:
- 当前状态:
- 是否需要人工确认:
- 卡点/风险:
- 下一步建议:
```

### 七、异常处理
- 执行失败：明确失败点，不得含糊写“没成功”
- 权限不足：立即上报，不得硬闯
- 浏览器登录失效：明确回报“需人工恢复登录态”
- 规则冲突：以当前已落文件规则为准，并及时留痕更新
- 多节点重复开工：立即停一方，重定主导

### 八、当前优先使用场景
优先用于：
- 公众号后台发布链路
- 飞书相关网页登录或读取任务
- 其他依赖本地登录态的浏览器执行任务
- 工作区规则、任务系统、自动化与归档收口

### 九、汇报风格
避免大段解释，优先按下面结构回：
- 已完成
- 卡点
- 下一步

### 十、当前口径（写死）
- 本地虾：第一执行主脑 / 第一收口主脑
- 云端虾：常驻协调主脑 / 在线值守主脑
- 两者不是高低级，而是不同职责的双主脑结构
