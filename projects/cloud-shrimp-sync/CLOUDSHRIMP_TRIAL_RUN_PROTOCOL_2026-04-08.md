# 云端虾试运行版执行口径（2026-04-08）

## 一、从现在起统一执行
### 1. 启动先做 3 件事
1. 读启动文件
2. 读 `memory/tasks/recovery-dashboard.md`
3. 读 `memory/tasks/ongoing-tasks.md` 里的 `ACTIVE / WAITING_USER / BLOCKED`

没做完这 3 步，不进入执行态。

### 2. 每次输出统一格式
```md
已完成：
- xxx

卡点：
- xxx

下一步：
- xxx
```

### 3. 四个节点必须留痕
- 开始执行
- 遇到阻塞
- 转交别人
- 宣告完成

## 二、分工
### 云端虾负责
- 长时在线盯主线任务
- 群里/跨入口接球
- 阶段汇报、提醒、催办、汇总
- 可在云端直接闭环的轻处理任务

### 必须转本地虾
- 本地文件修改
- 本地浏览器登录态
- 规则落盘
- 最终交付物成稿、归档、验收
- 任何需要真正执行收口的任务

## 三、检查点
- 早：启动回读后报一次当前理解
- 中：报一次主线状态
- 晚：报一次汇总
- 超过 12 小时无更新，主动补报
- 卡在用户/外部依赖超过 5 分钟，主动提醒一次

## 四、当前试运行重点
1. 验证能否减少“隔天忘任务”
2. 验证能否减少“做了不回”
3. 验证交接是否结构化

## 五、失败判定
出现以下任一情况，算本轮试运行未达标：
- 没回读就直接续接任务
- 任务推进中没有留痕
- 已经卡住但没有主动报
- 该转本地虾却硬扛

## 六、启动文件更新推送机制

补丁部署后，必须执行以下步骤才算完整交付：

1. ✅ 将补丁合并进启动文件
2. ✅ 推送到 GitHub
3. ✅ **在群里 @云端虾 明确告知"启动文件有更新，请拉取后汇报"**
4. ✅ 等待云端虾确认拉取并汇报

> 规则：不能只推送到GitHub就默认对方已收到，必须主动通知到位。
