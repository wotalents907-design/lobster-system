# 云端虾协作推进简报（2026-04-07）

## 一、现状
- 本地主脑负责主执行、主落盘、主验收、主汇总。
- 云端虾定位已明确，不是唯一主脑，而是常驻协调主脑、在线值守主脑。
- 当前真实问题不是“不会做”，而是三件事：隔天忘任务、做了不回、交接不结构化。
- 当前最适合云端虾承接的是：长时在线盯任务、跨入口接球、阶段汇报与催办。

## 二、当前分工

### 云端虾现在就承接的 3 个动作
1. **盯主线任务状态**
   - 重点盯：`TASK-CLOUDSHRIMP-BOOTFILE-001`、`TASK-MEMORY-RETRIEVAL-001`、`TASK-EVOLUTION-SYSTEM-001`
   - 每次启动先回读 recovery-dashboard 和 ongoing-tasks，再判断最新状态。

2. **做阶段汇报与催办**
   - 对超过半天没有更新的主线任务，主动输出一次三段式状态：已完成 / 卡点 / 下一步。
   - 如果卡在外部依赖或等待某方回传，云端虾负责提醒、催办、汇总。

3. **做跨入口接球与分流**
   - 群里、飞书、跨设备消息先由云端虾判断：是云端能直接闭环，还是要转本地虾执行。
   - 能云端直接做的轻处理任务，云端直接做；依赖本地权限/本地文件/本地浏览器的，马上转本地虾。

### 必须转回本地虾的边界
- 需要本地浏览器登录态
- 需要改本机文件或规则落盘
- 需要最终交付物成稿、归档、验收
- 需要真正收口完成、而不是只做跟踪提醒的执行型任务

## 三、固定节奏

### 启动节奏
每次新会话或中断恢复后，云端虾固定做 4 步：
1. 读启动文件
2. 读 recovery-dashboard
3. 读 ongoing-tasks 中 ACTIVE / WAITING_USER / BLOCKED 项
4. 先发一条“我当前理解”的状态摘要，再继续接球

### 汇报节奏
- **启动后**：立即报一次当前理解
- **中午**：报一次主线状态
- **晚间**：报一次汇总
- **遇到阻塞**：立刻报，不等
- **任务超 12 小时无更新**：主动补一次催办/汇报

### 固定汇报格式
```md
已完成：
- xxx

卡点：
- xxx

下一步：
- xxx
```

## 四、最小修补措施（解决失忆 / 失联 / 假推进）
1. **强制启动回读**
   - 不允许云端虾一开场直接续写任务，必须先回读启动文件 + recovery-dashboard + ongoing-tasks。
   - 如果没读，不进入执行态。

2. **强制阶段留痕**
   - 任务开始、阻塞、交接、完成，至少四个节点必须留一条结构化更新。
   - 禁止“后台在弄”但没有任何状态更新。

3. **强制交接模板**
   - 转本地虾或对用户汇报时，必须包含：任务ID、当前状态、卡点、下一步、是否需人工确认。
   - 禁止口头派活、禁止模糊说“你接着看”。

## 五、下一步
1. 把“启动回读 + 三段式汇报 + 四节点留痕”正式并入云端虾启动文件。
2. 从明天开始，按早/中/晚三个检查点试运行一轮。
3. 若连续两轮仍出现“有做没回”，直接把该问题升为事故继续追踪。

## 六、我建议立刻执行的一条
**先把“每次启动必须先回读 recovery-dashboard 和 ongoing-tasks，再发一条三段式当前理解”定成硬规则。**

这条一落地，能先把“隔天忘任务”和“做了不回”一起压下去大半。
