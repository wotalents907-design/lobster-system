# 多虾统一文件骨架标准_V1

## 目标

给本地虾、云端虾、秘书虾建立一套统一的 rules / memory / task 文件骨架。
目标不是让所有内容一模一样，而是：
- 入口一致
- 恢复路径一致
- 协作协议一致
- 角色内容可分化

---

## 一、必须统一的文件名 / 路径

以下文件建议所有虾都使用相同名称与相同相对路径：

### 顶层核心文件
- `AGENTS.md`
- `SOUL.md`
- `USER.md`
- `MEMORY.md`
- `TOOLS.md`
- `HEARTBEAT.md`

### 规则目录
- `rules/CONSTANTS.md`
- `rules/MEMORY_SCHEMA.md`
- `rules/MODEL_SWITCH_CHECKLIST.md`

### 记忆目录
- `memory/YYYY-MM-DD.md`
- `memory/tasks/ongoing-tasks.md`
- `memory/tasks/recovery-dashboard.md`

### 推荐保留的结构化目录
- `memory/profiles/`
- `memory/projects/`
- `memory/index/`
- `memory/inbox/`

---

## 二、为什么这些必须统一

### 1. 方便跨模型 / 跨环境恢复
只要文件入口一致，新模型、新 session、新节点都知道先读哪里。

### 2. 方便多虾交接
本地虾、云端虾、秘书虾在引用规则和任务面时，不需要重新翻译文件结构。

### 3. 方便以后自动化
后续无论做同步脚本、巡逻脚本、任务汇总脚本，统一路径都更容易做。

---

## 三、哪些可以差异化

统一的是“骨架”，不是“内容必须逐字一样”。

### 允许差异化的内容
#### 1. `SOUL.md`
可因角色不同而不同：
- 本地虾：执行主脑 / 收口主脑
- 云端虾：值守协调 / 巡逻主脑
- 秘书虾：前台接球 / 群协作协调位

#### 2. `HEARTBEAT.md`
可因角色不同而不同：
- 云端虾：巡逻 `TASK_TRACKER`
- 本地虾：可做本地恢复、任务检查、仓库收口
- 秘书虾：可做前台消息筛查与提醒

#### 3. `TOOLS.md`
环境相关内容本来就不同。

#### 4. `MEMORY.md`
长期偏好、业务规则可大体一致；但节点专属环境知识、角色边界可不同。

---

## 四、哪些内容必须保持一致

以下内容建议多虾之间尽量同步：

### 1. 用户长期偏好
例如：
- 永久中文回复
- 简洁、直接、少废话
- 少追问，多行动
- 需要选择时给结构化选项

### 2. 核心协作规则
例如：
- 不裸切模型，先收口
- 主动识别任务触发
- 有变化才汇报，无变化不刷屏
- 多代理协作时要有职责边界和验收格式

### 3. 任务状态协议
统一使用：
- `NEW`
- `ACTIVE`
- `WAITING_USER`
- `BLOCKED`
- `DONE`
- `CANCELLED`

### 4. 任务恢复顺序
统一先看：
- `recovery-dashboard`
- `ongoing-tasks`
- 近两天 daily memory
- 长期 memory

---

## 五、建议的分层规则

### A. 全局统一层
所有虾尽量一致：
- `rules/CONSTANTS.md`
- `rules/MEMORY_SCHEMA.md`
- `rules/MODEL_SWITCH_CHECKLIST.md`
- 用户长期偏好
- 任务状态协议

### B. 节点角色层
按虾种分化：
- `SOUL.md`
- `HEARTBEAT.md`
- 节点专属 `TOOLS.md`
- 节点专属长期记忆补充

### C. 项目任务层
按当前任务共享或分工：
- `memory/tasks/ongoing-tasks.md`
- `memory/tasks/recovery-dashboard.md`
- `memory/projects/*.md`

---

## 六、当前最推荐的执行方式

### 方案：统一骨架 + 差异化内容

也就是：
- 名字尽量统一
- 目录尽量统一
- 规则协议尽量统一
- 角色内容按职责分化

这是当前最稳、最不容易混乱的方案。

---

## 七、一句话原则

**文件骨架统一，角色内容分化，任务协议一致，恢复路径固定。**
