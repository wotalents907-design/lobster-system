# 本地补强 → 云端同步执行顺序表

> 目标：先把本地虾需要的关键能力、规则、分工与工作流补齐，再把成熟版本同步给云端，避免把半成品同步过去。

---

## 一、总原则

1. 先补本地，再同步云端。
2. 先打磨成熟版本，再做云端挂载。
3. 不把口头想法、试验性口径、未稳定分工直接同步给云端。
4. 只有已经清晰、可执行、可复用的内容，才进入云端同步包。

---

## 二、执行顺序

### 第一步：补本地虾关键缺口
优先补这几项：
1. 飞书文档 / wiki / 知识库协同能力
2. 浏览器 / 网页执行协同能力
3. 模型路由与升级判断纪律
4. 三虾（秘书虾 / 云端虾 / 本地虾）分工规则

### 第二步：把本地实践压成规则文件
补强过程中，逐步形成这些文件或内容：
- 分工文件
- 模型路由规则
- 飞书文档协作流程
- 交接模板
- 当前主线任务状态
- 例外情况与备援工作流

### 第三步：先在本地跑顺
判断标准：
- 能否稳定按规则推进
- 能否减少反复解释
- 能否减少掉球
- 能否减少模型乱切
- 能否在真实任务里复用

### 第四步：筛选可同步内容
可同步给云端的，应优先是：
- 角色分工
- 模型路由规则
- 交接规则
- 记忆规则
- 当前主线最小状态

不应优先同步的：
- daily 原始流水
- 尚未稳定的试验口径
- 过细历史过程
- 还在频繁变动的局部策略

### 第五步：同步给云端
同步后，云端应把这些当成：
- 挂载文件
- 切模型后优先回读文件
- 协调与分流依据
而不是当成一次性阅读材料。

### 第六步：观察云端执行效果
观察指标：
- 切模型后失忆是否减轻
- 分流是否更合理
- 模型使用是否更省钱
- 是否减少重复问用户
- 是否减少与本地虾扯皮或重复开工

### 第七步：再做第二轮修订
如果云端执行后暴露问题，再回到本地收口修订，形成第二版成熟规则后再继续同步。

---

## 三、一句话执行口径

先把本地虾补成成熟版，再把成熟版同步给云端；不要把半成品和临时想法直接同步过去。
