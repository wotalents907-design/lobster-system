# feishu-chatrecord-ocr

> 微信委托聊天记录图片OCR识别与飞书文档整理

## 触发条件

用户发送委托聊天记录截图并要求整理到飞书文档时触发。

## 核心原则（必须遵守）

### 完整整理不筛选原则
处理聊天记录图片中的案例时，**必须将所有案例全部整理，不应自行筛选或遗漏任何案例**。每一个独立案例都要提取，不得以"重复"、"类似"为由合并或跳过。

### 原始对话逐字输出格式
输出原始聊天记录时逐字呈现，不提炼不加功，格式为：

```
[日期时间] 当事人/律师：原始对话内容
```

如有错别字可适当修改。

### 文档命名规则
文档名称按「**来源-日期-当事人**」统一编码：

| 来源编码 | 含义 |
|----------|------|
| A | 抖音 |
| B | 视频号 |
| E | 其他 |

日期格式：`26/4/7`（年份/月份/日期，不补零）

示例：`A-26/4/7-许亮`

## 工作流

### 第一步：OCR识别
1. 接收用户发送的聊天记录图片（通常5张一批）
2. 使用 Tesseract OCR 或 `pdf-processing` skill 进行文字识别
3. **处理超窄长图**：若图片宽高比异常（宽度 < 300px 且高度 > 1500px），先切分为多个区域再识别

### 第二步：提取案例信息
从OCR文本中提取：
- 日期时间（每条消息的时间戳）
- 当事人名称（律师/客户）
- 对话内容（逐字）
- 独立案例数量

### 第三步：写入飞书文档
1. 在飞书知识库或指定文件夹创建文档
2. 文档标题：「来源-日期-当事人」
3. 文档内容按「[日期时间] 当事人/律师：内容」格式逐条输出

### 第四步：核查完整性
- 检查是否有遗漏案例
- 对照用户提供的图片数量，核对提取的案例数量是否完整
- 如有遗漏，主动补录

## 编码冲突说明

**注意**：用户销转流程中已使用ABCD对客户分等级（A/B/C/D四类），与来源编码（A=抖音/B=视频号/E=其他）**是两套独立体系**，不得混淆。

- **来源编码**（本skill用）：A/B/E
- **销转等级**（用户分级体系）：A/B/C/D

## 典型对话

用户：「整理一下这些聊天记录」+ 发送图片

→ 执行本skill，输出：「已完成：[X]个案例已整理到飞书文档：[文档链接]」

## 依赖工具

- `feishu_doc` / `feishu_wiki` — 飞书文档写入
- `pdf-processing` — OCR文字识别
- `exec` — 图片预处理（切图、缩放）
