## 二、给云端虾补 Git，目标不是花架子

目标是让云端虾后续能稳定完成这几类动作：
- 拉取最新规则/状态
- 提交规则更新
- 推送云端同步材料
- 给云端工作流留痕
- 避免本地文件改了一堆却没有版本回滚点

不是为了“看起来专业”去配 Git，
而是为了让双虾/三虾协作、规则演进、云端同步包维护有可回滚、可对比、可交接的底座。

---

## 三、当前还缺什么

### 1. Git 身份
至少要补：
- `git config user.name`
- `git config user.email`

### 2. 远程仓库
至少要明确：
- GitHub / Gitee / 私有 Git 仓库地址
- 后续是 HTTPS 还是 SSH
- 谁负责首次建仓

### 3. 使用口径
必须先定：
- 哪些文件允许直接提交
- 哪些文件只本地保留
- 是否允许把 `memory/` 全量上云
- 是否允许把含敏感配置的文件推送到远程

---

## 四、我建议的 Git 口径

### 推荐用途
适合上 Git 的：
- 规则文件
- 角色文件
- 交接文件
- 云端同步包
- TODO / 任务结构文件
- 项目规划与 SOP

### 不建议直接上 Git 的
需先审查再决定是否提交：
- 含账号密钥/敏感 token 的配置文件
- 高隐私 daily memory 原文
- 含登录截图、验证码、隐私材料的文件
- 可能暴露客户/案件细节的原始资料

### 推荐策略
把 Git 先当作：
**规则层 / 协作层 / 结构层的版本管理底座**
而不是把整个工作区不加筛选地全量推上去。

---

## 五、给云端虾补 Git 的推荐顺序

### 第一步：补身份
配置本仓库级：
- `user.name`
- `user.email`

### 第二步：定远程
补 `origin`

### 第三步：先做一次干净首提
优先提交：
- `ROLE_CLOUD.md`
- `ROLE_LOCAL.md`
- `HANDOFF.md`
- `CLOUD_SYNC_PACK.md`
- `TODO.md`
- `工作规划/云端小龙虾规则包.md`
- `云端虾启动文件_飞书云文档版.md`

### 第四步：再定 memory / projects / skills 的上云边界
避免第一次就把高噪音、高敏感目录全推上去。

---

## 六、我最推荐的实际做法

### 方案 A（最稳）
先把当前仓库作为**本地工作母仓**，只把“规则层 + 协作层 + 规划层”接上远程。

优点：
- 风险最低
- 最适合云端虾维护
- 不容易把 daily/隐私材料全推上去

### 方案 B（不推荐立刻做）
整个 workspace 直接全量接远程并推送。

风险：
- 噪音大
- 敏感文件容易误上云
- 后续清理成本高
