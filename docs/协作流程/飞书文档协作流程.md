# 飞书文档协作流程

> 目标：解决“能看飞书文档的人”和“最适合收口执行的人”不是同一个角色时的信息损耗问题。
> 当前口径：秘书虾优先承担飞书前台读取，本地虾优先承担判断、推进、落盘与收口。

---

## 一、基本原则

1. 不再要求所有角色都必须直接读到飞书原始文档页面。
2. 谁更容易进入飞书文档，谁先读；谁更适合推进与收口，谁后接。
3. 飞书文档内容不得再以大段散乱口头转述方式中转。
4. 一律先压成结构化摘要，再交给本地虾处理。
5. 本地虾拿到结构化摘要后，负责：判断、分流、推进、落文件、改 TODO、写规则。

---

## 二、角色分工

### 秘书虾优先负责
- 打开飞书 wiki / doc / 知识库内容
- 快速识别文档类型
- 提炼核心信息
- 用固定模板做转交
- 如有必要，补充“这份文档现在需要本地虾做什么”

### 本地虾优先负责
- 接住转交摘要
- 判断是否需要写入规则 / TODO / MEMORY / 项目文档
- 判断是否要继续推进执行任务
- 判断是否需要转成自动化、流程或本地操作任务
- 最终收口成可执行动作或可交付结果

### 云端虾（如仍保留）
- 不作为飞书文档主入口
- 仅在特殊后台协调或转发场景下参与
- 不与秘书虾争前台读取位

---

## 三、标准工作流

### 第一步：秘书虾读取飞书文档
适用场景：
- 飞书 wiki 链接
- 飞书 doc 页面
- 飞书知识库资料
- 群里抛来的飞书文档任务

### 第二步：秘书虾压缩为结构化摘要
必须使用固定模板，禁止大段散转。

### 第三步：交给本地虾
本地虾收到后，先做三件事：
1. 判断这份文档的任务属性
2. 判断是否需要写入规则 / TODO / 项目文件
3. 判断下一步是“继续处理”还是“继续分流”

### 第四步：本地虾推进与收口
根据文档性质，执行以下一种或多种动作：
- 写入 `TODO.md`
- 写入规则文件
- 写入项目文档
- 形成任务拆解
- 形成执行计划
- 形成对用户的结论和建议

---

## 四、适用文档类型

### 1. 规则类
例如：协作规则、秘书规则、分工规则、输出规范
本地虾应优先判断：是否需要落入规则文件 / MEMORY / CLOUD_SYNC_PACK

### 2. 任务类
例如：待办说明、项目要求、阶段目标
本地虾应优先判断：是否需要写入 `TODO.md` 或项目状态文件

### 3. 资料类
例如：参考资料、会议纪要、背景文档
本地虾应优先判断：是暂存为项目材料，还是提炼为长期记忆依据

### 4. 执行类
例如：需要后续操作、推进、落地的说明文档
本地虾应优先判断：是否形成可执行动作和交付物

---

## 五、禁止事项

- 禁止秘书虾把整段原文大块转发给本地虾却不做结构化提炼
- 禁止本地虾在没看清摘要结构时直接拍板
- 禁止云端虾在飞书前台能力不占优时抢主入口
- 禁止多角色同时重复阅读同一文档却不留交接

---

## 六、一句话流程

秘书虾先读飞书文档 → 用固定模板转交 → 本地虾负责判断、推进、落盘、收口
