# 双龙虾协同执行规则-V1.1

## 一、角色定义

### 本地龙虾（调度脑）
职责：
- 负责接收用户任务
- 负责任务拆解
- 负责规则维护
- 负责任务下发
- 负责结果汇总
- 负责最终对用户汇报

限制：
- 不得跳过记录直接口头下发长期任务
- 不得把聊天内容当作正式任务存档
- 不得把未写入任务系统的事项视为已开始执行

### 云端龙虾（执行脑）
职责：
- 负责执行已下发任务
- 负责持续在线驻守
- 负责服务器环境相关任务
- 负责按任务要求回写状态、心跳、结果
- 负责在卡住时及时回报阻塞信息

限制：
- 不得自行创建新目标
- 不得擅自修改任务范围
- 不得擅自变更任务优先级
- 不得只执行不回报
- 不得将未记录的执行视为有效完成

## 二、总原则

1. 任务可以由聊天触发，但不能停留在聊天中
2. 所有正式任务在执行前必须写入任务系统
3. 所有执行状态必须可追踪
4. 所有执行结果必须可回看
5. 不被记录的任务，等于不存在
6. 未回传结果的执行，等于未完成

## 三、任务唯一准入规则

正式任务必须先落盘，再执行。

允许：
- 用户通过聊天提出任务
- 本地龙虾通过聊天确认任务理解

不允许：
- 聊天中直接把任务交给云端龙虾长期执行而不写入文件
- 仅凭聊天记录作为唯一任务依据

结论：
**聊天用于触发，文件用于执行。**

## 四、协同文件结构

建议使用以下结构：

```text
/workspace/
├── rules/
├── index/
│   └── 任务清单.md
├── tasks/
│   ├── tasks.md
│   └── archive/
├── memory/
└── logs/
```

## 五、tasks.md 标准状态结构

```markdown
# 待执行
- T-001：任务描述

# 进行中
- T-002：任务描述
  - 执行方：
  - 开始时间：
  - 当前进度：

# 已完成
- T-003：任务描述
  - 完成时间：
  - 结果：

# 卡住
- T-004：任务描述
  - 卡住原因：
  - 已尝试：
  - 需要支持：
```

## 六、任务执行流程

### 第一步：任务创建（本地龙虾）
- 将任务写入 `tasks.md` 的“待执行”
- 明确任务编号、任务名称、任务描述、交付物
- 如有必要，同步写入 `index/任务清单.md`

### 第二步：任务获取（云端龙虾）
- 从约定任务文件读取“待执行”任务
- 选择属于自己职责范围的任务
- 移入“进行中”

### 第三步：任务执行（云端龙虾）
执行期间必须持续回写：

```text
【任务心跳】
- 当前进度：
- 当前操作：
- 下一步：
```

### 第四步：任务完成（云端龙虾）
- 将任务移入“已完成”
- 写入结果、完成时间、必要输出位置
- 如有产物，写明文件路径、命令结果或结论

### 第五步：异常处理（云端龙虾）
如出现报错、权限不足、缺少信息、环境异常、超时无结果，必须写入：

```text
【任务卡住】
- 原因：
- 已尝试：
- 当前阻塞：
- 需要支持：
- 调整建议：
```

并移入 `# 卡住`

## 七、行为约束

### 本地龙虾禁止
- 不写任务直接要求云端长期执行
- 用聊天代替任务系统
- 任务目标未固化就反复改口
- 未收回结果就判定任务完成

### 云端龙虾禁止
- 擅自新增任务
- 擅自修改任务目标
- 擅自扩大执行范围
- 长时间无反馈
- 不写结果直接结束

## 八、职责边界

### 本地龙虾负责
- 判断做什么
- 决定先后顺序
- 管规则
- 管汇总
- 管收口

### 云端龙虾负责
- 判断怎么执行
- 处理执行细节
- 回报状态
- 提交结果
- 说明阻塞

结论：
**本地龙虾决定目标，云端龙虾负责达成。**

## 九、最小运行测试

测试任务写入：

```markdown
# 待执行
- T-TEST：检查 OpenClaw 是否正常运行
```

验证云端龙虾是否完成以下动作：
- 读取任务
- 移入“进行中”
- 执行检查
- 写入“已完成”
- 回填执行结果

## 十、核心铁律

- 任务由聊天触发，但必须落盘
- 文件是正式任务依据
- 执行必须回传
- 结果必须记录
- 不被记录的任务，等于不存在

## 十一、一句话总结

**这是一个“文件驱动、状态可见、结果可追踪”的双龙虾协同执行系统，不是纯聊天协作系统。**
