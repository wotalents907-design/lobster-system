# 多代理一致性控制框架

> 基于 2026-04-05 讨论 + Mac合伙人分析 整合

---

## 三层影响模型

| 层次 | 决定因素 | 影响什么 |
|------|----------|----------|
| **第一层：模型层** | 底层大模型 | 理解方式、推理风格、对模糊指令的补全 |
| **第二层：规则层** | AGENTS.md / MEMORY.md / CONSTANTS | 什么必须做、什么不能做、状态枚举 |
| **第三层：Soul层** | SOUL.md | 说话风格、主动性、表达方式 |

---

## 一致性控制优先级

**目标：减少「同一命令不同反应」，重点改这些：**

### 🔴 最高优先：统一硬规则（比 Soul 更硬）

必须统一的（强制执行，写入 rules/CONSTANTS.md）：
- 任务状态枚举：NEW / ACTIVE / WAITING_USER / BLOCKED / REVIEWING / DONE / ARCHIVED
- 授权边界：内部升级无需询问，外部操作必须确认
- 记忆格式：所有 entry 必须加结构化标签

### 🟡 次优先：统一记忆底座

- 所有同类代理启动时读取同一套 rules/
- 重要决策必须写 DECIDED_BY 记录
- TASK 状态变化必须同步到 memory/

### 🟢 第三优先：统一职责边界

| 角色 | 职责 |
|------|------|
| 秘书虾子 | 协作中枢、记忆管理、日常响应 |
| Mac合伙人 | 技术研究、源码分析 |

---

## 场景权重排序

**场景A：命令模糊（"你们看着办"）**
权重：模型差异 > 记忆差异 > Soul 差异

**场景B：命令清晰（"总结三点发我"）**
权重：Soul 差异 > 模型差异 > 职责差异

**场景C：多代理分工明确**
权重：职责设定 > 记忆调用 > 模型差异 > Soul 差异

---

## 协作规则

1. 任务认领后必须更新 TASK_TRACKER.md
2. 完成/阻塞时必须通知主人
3. 新 session 启动先读 TASK_TRACKER.md
4. 自主升级完成后写备忘录到 memory/
